Pre-deployment technical audit

RoboSub 2026 AUV Software Stack

Sensors → perception → planning → control → Pixhawk → thrusters, traced end-to-end.

Audited on the vehicle (Jetson, ~/robosub2026/robosub-2026) · 2026-07-12 · ~15k lines project code across root field scripts + 7 ROS 2 packages (vendored ZED/BehaviorTree.CPP packages skimmed, not audited)

1 · Executive summary

The single most urgent problem is not a code bug: this vehicle's checkout is one commit behind origin/main, and that missing commit (3ac71152, pushed 2026-07-12 by a teammate) contains the competition gate mission (gate_task.py), the DepthKeeper depth-hold engine, the detector letterbox/BGRA fix, and the 6-DOF message/thruster changes. The untracked gate_spin_pass.py on this machine imports all of it — it crashes with ModuleNotFoundError before arming anything. Until you pull and rebuild auv_msgs, the primary mission script cannot run and the detector on disk still feeds the model channel-swapped, aspect-squashed frames.

Beyond the sync issue, the stack is really two parallel autonomy stacks: the field-proven Python mission scripts (field_common / DepthKeeper / gate_task) and a C++ behavior-tree scaffold (SHRUB) whose motion primitives are open-loop timed stubs, whose manipulation nodes are TODOs (the working dropper.py driver is not wired in), and whose blackboard defaults are success-biased. Only the Python gate stack is plausibly competition-ready.

The most dangerous runtime behaviors found: the thruster node auto-re-arms after any disarm (including failsafe or operator disarm); several closed-loop rotation routines integrate the absolute value of angle deltas, so sensor noise and rocking count as rotation progress; the Bar02 depth-hold test has no stale-depth detection, which reproduces the observed "Bar02 drops → ArduSub forces MANUAL → sub sinks while the script trusts a frozen reading" failure; and multiple paths fall back to a silent simulation mode mid-mission, leaving the stack "running" with dead thrusters.

Do these before the next water session, in order:
  1. git stash or commit the local edits to depth_hold_bar02_test.py / motor_test.py, then git pull. Expect conflicts in both files — origin also changed motor_test.py (switched to MOTOR_TEST_THROTTLE_PWM because ArduSub 4.5.7 rejected percent type) while the local copy was retuned on 2026-07-12. Merge deliberately; both sides contain fixes.
  2. Rebuild generated messages: colcon build --symlink-install --packages-select auv_msgs (then dependents). Without this, DepthKeeper's pitch_rate/roll_rate fields don't exist → its control thread dies on the first tick (F2).
  3. Re-run gate_spin_pass.py --dry-run and a bench gate_task.py --no-style --yes-less smoke test to confirm imports, detector letterbox path, and Bar02 detection.

2 · Architecture assessment

Layout: root-level field scripts (mission runners + bench tools) over ROS 2 packages — auv_msgs, mavlink_thruster_control (ThrusterController + SafetyMonitor), vision (front detector + bottom camera), localization (fusion, depth, drift correction, a disabled VSLAM node), control (autonomous_controller + centering framework), prequalification, and bt_mission (C++ BehaviorTree.CPP executor). Workspace is built with --symlink-install (egg-links point at src/), so Python edits take effect without rebuilds — but message/C++ changes do not.

  • Duplication is the main structural risk. Four generations of mission logic coexist: gate_runner.py (legacy, STABILIZE, owns its own MAVLink link), prequalification_node, the Python gate stack (gate_task/gate_spin_pass on origin), and the SHRUB BT. Running the wrong one at the dock is a real failure mode; the legacy runner even uses a different flight mode and depth strategy.
  • Two control authorities on one topic. movement_command is published by RampedDriver, DepthKeeper, autonomous_controller, prequal node, and MissionIO. The single-writer discipline exists only by convention (documented in field_common); nothing enforces it, and the BT + autonomous_controller + a field script running together would fight at 10 Hz.
  • Frame conventions are split. ZED world is RIGHT_HANDED_Y_UP (vertical = Y); field_common correctly computes heading about Y, but localization_node, autonomous_controller, and MissionIO extract yaw with the Z-up formula and splice depth into z. Everything downstream of localization/pose (station_keep, waypoint) is frame-inconsistent (F18).
  • Good patterns worth keeping: the DepthKeeper single-writer engine, the Bar02 sanity gate (30BA misdetect check), the ZED open/reopen-with-retry loops, the passive master.messages heartbeat read under an external recv owner, and the pymavlink add_message monkeypatch.

Ranked by severity. File paths are relative to the repo root; line numbers refer to the working tree unless marked origin (the unpulled commit 3ac71152).

F1 · Vehicle checkout is behind origin/main; the gate mission script cannot even import

CriticalOps / Deploymentgate_spin_pass.py:69Confidence: High
Description
Local main is at a440c08e; origin/main is at 3ac71152 ("Add gate task missions, DepthKeeper depth-hold engine, motor trim tool"). The untracked gate_spin_pass.py does from gate_task import dive, dive_open_loop, search_gate and uses fc.Bar02DepthSource, fc.DepthKeeper, fc.clamp — none of which exist in the checked-out field_common.py, and gate_task.py is not on disk at all.

Failure scenario
At the dock: python3 gate_spin_pass.py → ModuleNotFoundError: No module named 'gate_task'. Zero mission capability until fixed. Worse: a panicked partial fix (copying one file) would run new mission code against the old field_common/thruster/msg stack.

Merge both sides by hand, then commit and push so the two machines converge.

F2 · After the pull, stale generated messages silently kill the DepthKeeper thread

CriticalROS / Buildfield_common.py (origin) · src/auv_msgs/msg/MovementCommand.msgConfidence: High
Description
Origin adds pitch_rate/roll_rate to MovementCommand; DepthKeeper's _send() sets them every tick. Python packages are symlink-installed (live), but auv_msgs is generated code in install/ — it does not update on pull.
Root cause
Symlink-install gives the illusion that pulling is enough; message and C++ packages still need colcon build. (Already a documented trap for this repo.)
Failure scenario
Pull without rebuild → first keeper tick raises AttributeError: pitch_rate inside the daemon control thread. The thread dies silently (no guard, F21); the thruster node holds the last commanded axes for up to 30 s (watchdog default) with no depth-hold heave, then stops but stays armed in ALT_HOLD at depth.
Recommended fix
colcon build --symlink-install --packages-select auv_msgs mavlink_thruster_control bt_mission after every pull that touches msg/ or C++. Add a startup assertion in field_common: assert hasattr(MovementCommand(), 'pitch_rate') with a "rebuild auv_msgs" message.

F3 · Detector on disk feeds the model channel-swapped, aspect-squashed frames

CriticalVisionsrc/vision/vision/detector.py:656,115,301Confidence: High
Description
The working-tree inference thread converts ZED frames with COLOR_RGBA2RGB, but ZED get_data() returns BGRA — red/blue are swapped. Preprocessing is a plain cv2.resize of 1280×720 → 416×416 (~1.8× horizontal squash the model never saw in training). Both are fixed in the unpulled commit (BGRA2RGB + letterbox with gain/pad un-mapping).
Root cause

Failure scenario
Every mission run on this machine until the pull: depressed confidences, missed gate/marker detections at range, mis-localized boxes — exactly the degraded-detection behavior chased on 2026-07-10.
Recommended fix
Pull (F1). Then also fix the same two bugs in bottom_camera_node.py, which origin did not touch (F12).

F4 · Depth-hold test trusts a frozen depth reading — reproduces the "Bar02 drops, sub sinks" incident

CriticalSafety / Sensorsdepth_hold_bar02_test.py:731–830 (main loop, drain_depth)Confidence: High
Description
drain_depth() returns None when no fresh pressure message is buffered, and the loop then keeps using the last depth value forever. There is no staleness timer, and the loop never watches heartbeat custom_mode after the initial ALT_HOLD verification.
Root cause
Depth freshness was only considered at setup (detection + surface latch), not during the run. The Bar02's I2C connection is known-intermittent on this vehicle.
Failure scenario
Bar02 drops mid-run at 0.5 m with target 0.9 m: ArduSub silently forces MANUAL (mode 19), where the script's z < 500 "descend" frames become direct down-thrust. The P controller sees a frozen 0.5 m error and commands descent indefinitely; the --max-depth abort can never trip because the reading never changes. This is the observed 2026-07-09 sink signature. The abort/surface paths have the same hole (loop until frozen depth ≤ deadband → never exits until the 30 s surfacing deadline or Ctrl-C).
Recommended fix
Track last_depth_time; if no pressure message for >1.5 s → declare depth lost, send neutral z, surface on timer, disarm. Also re-check HEARTBEAT.custom_mode each second and treat an unexpected exit from ALT_HOLD as an abort. (DepthKeeper on origin already handles staleness via depth(stale_s=2) — the standalone test should match.)

F5 · Thruster node auto-re-arms after any disarm, including failsafe or operator disarm

HighSafety / Pixhawksrc/mavlink_thruster_control/…/thruster_node.py:316–329Confidence: High
Description
_check_armed_status() treats every disarm as an anomaly: it re-sends the flight mode and re-arms, forever, with no attempt counter and no way to distinguish "ArduSub battery/leak failsafe disarmed us" or "a human disarmed us from QGC / tether" from a genuine glitch.
Root cause
Reconnect robustness written without an operator-intent model.
Failure scenario
Poolside: diver near the vehicle, operator disarms from QGC — five seconds later the node re-arms and props can spin. Or: ArduSub disarms on a battery failsafe and the node re-arms a brownout-prone vehicle repeatedly.
Recommended fix
Re-arm only if the node itself has not sent a disarm, cap consecutive re-arms (e.g. 3), and respect failsafe disarms: parse the STATUSTEXT that accompanies the disarm; if it contains "failsafe"/"leak"/"battery", stop re-arming and latch a critical flag instead.

F6 · Silent "simulation mode" fallbacks let a mission keep running with dead thrusters

HighSafety / Pixhawkthruster_node.py:197–199, 248–254Confidence: High
Description
If no serial device answers at startup, or 5 reconnect attempts fail mid-run, the node sets simulate = True and continues as if healthy. Mission scripts, the BT, and the safety monitor all keep "succeeding" against a vehicle that is producing no thrust — or, mid-water, one that just went limp.
Root cause
A desk-development convenience left in the competition path. (Memory notes the same symptom from the pymavlink add_message crash: "SIMULATION mode" was the visible symptom of a dead recv path.)
Failure scenario
USB hiccup during a run → 5 failed reconnects → vehicle drifts/floats with the whole stack convinced the mission is progressing. At the surface before launch: wrong port → everything "works", nothing moves, five minutes of run window burned.
Recommended fix
Make simulate opt-in only (simulate:=true); on hardware, connection loss should escalate loudly — publish a /safety/thruster_link_lost Bool the BT maps to critical_failure, and have field_common's session abort if thrusters.simulate flips true.

F7 · Every closed-loop rotation integrates |Δangle| — noise and rocking count as progress, direction is never checked

HighNavigation / Controlfield_common.py:319 · origin DepthKeeper.turn/rotate · gate_spin_pass.py:301Confidence: High
Description
turned += abs(wrap(h - h_prev)) (ZED path) and rotated += abs(rates[idx]) * dt (gyro path) accumulate unsigned increments. Heading jitter, current-induced oscillation, and — worst — ALT_HOLD/self-leveling fighting a roll all advance the counter.
Root cause
Chosen to be "immune to sign conventions", which also makes it immune to being wrong.
Failure scenario
Style points: in MANUAL, a roll attempt that stalls at ±25° and rocks (the exact 2026-07-10 behavior at 31°) accumulates |p|·dt until the 90° increment "verifies" — the script reports a scored roll that never happened, ends the entry with the sub tilted, then re-latches depth while badly oriented. Turns: a noisy 10 Hz ZED heading adds phantom degrees, so 90° turns systematically under-rotate; the gate re-aim masks it until vision drops.
Recommended fix
Integrate the signed delta projected on the commanded direction: turned += max(0, direction * wrap(h - h_prev)) (equivalently direction * rates[idx] * dt), and declare failure if net progress stalls below a rate floor for ~2 s instead of silently completing.

F8 · ZED pose/depth published without checking tracking state — frozen odometry defeats every staleness check

HighSensors / ZEDsrc/vision/vision/detector.py:961–970 · localization/vslam_node.py:189Confidence: High
Description
run_detector() calls zed.get_position() and publishes depth/sub_depth + vslam/odometry every frame, ignoring the returned POSITIONAL_TRACKING_STATE. When tracking is SEARCHING/OFF the last pose keeps streaming with fresh timestamps. vslam_node.py has the same hole. (bottom_camera_node.py checks the state — the only one that does.)
Root cause
Return value ignored.
Failure scenario
Tracking drops during a spin (VSLAM loses lock after style moves — a known pattern here): HeadingMonitor keeps returning a frozen heading, so DepthKeeper.turn sees zero deltas and burns its whole timeout mid-rotation while the vehicle keeps yawing; CoordMonitor's have_fix() stays true, so gate_spin_pass's cross-track law corrects against a stale position; ZED-derived depth freezes while the sub moves vertically.
Recommended fix
Publish pose/depth only when state == OK; on any other state, stop publishing (staleness monitors downstream already do the right thing the moment messages stop).

F9 · autonomous_controller leaves stale thrust on non-dominant axes — station-keep and waypoint spiral

HighThrusters / Controlsrc/control/control/autonomous_controller.py:665–686Confidence: High
Description
_dispatch_axes() sends only the single largest axis per tick as a verb (rotate_cw, surge_forward, …). Thruster verbs only touch their own axis: once rotate_cw 0.5 has been sent, current_r stays 500-scaled until a stop — subsequent surge_forward ticks never clear it.
Root cause
Dominant-axis dispatch bolted onto single-axis verb semantics; the correct multi-axis path (command='axes') exists and is used by track_object, but not here.
Failure scenario
waypoint mode: the "turn first if off heading" phase commands yaw; once heading closes, surge becomes dominant — but the last yaw command keeps spinning the sub at 10 Hz forever. The vehicle corkscrews and never reaches the waypoint; station_keep oscillates the same way.
Recommended fix
Replace _dispatch_axes with _dispatch_setpoint(surge, strafe, heave, yaw) — one 'axes' message with all four values each tick.

F10 · DepthKeeper heave commands below effort 0.2 are eaten by ALT_HOLD's stick deadzone

HighThrusters / Pixhawkthruster_node.py set_axes (origin) · field_common.py DepthKeeper (origin)Confidence: High
Description
set_axes() maps heave linearly: effort 0.15 → z = 425, only 75 counts from neutral — inside ArduSub's THR_DZ (±100), which ALT_HOLD treats as "hold depth". The standalone depth test learned this (its vertical_z() offsets past THR_DZ; fixed 2026-07-12), but the DepthKeeper→thruster path never got the same mapping. Default min_speed=0.15 is therefore a no-op in ALT_HOLD, and the keeper's "always-active P counters buoyancy inside the deadband" only actually works in MANUAL.
Root cause
The deadzone lesson was applied to one code path, not the shared one.
Failure scenario
In ALT_HOLD the keeper believes it is nudging depth with 0.1–0.2 efforts while ArduSub ignores them; depth error must grow past ~0.1 m (kp=2) before any command bites, producing a sawtooth around the target. If ALT_HOLD is rejected (Bar02 gone → MANUAL), the same small efforts suddenly become real thrust — the controller's effective gain changes 0→full with flight mode.
Recommended fix
In set_axes(), offset non-zero heave past the deadzone exactly like vertical_z(): z = 500 − sign(h)·(THR_DZ + |h|·(500−THR_DZ)), and read THR_DZ from params at connect instead of hardcoding.

F11 · Mission timeout and mission end leave the vehicle armed at depth; nothing in the ROS stack can disarm

HighSafety / BTsrc/robosub2026/src/bt_executor.cpp:129–201Confidence: High
Description
On the 14.5-minute timeout the executor halts the tree and sends stop — z neutral, ALT_HOLD holds current depth, vehicle stays armed and submerged. There is no surface-then-disarm epilogue, and no ROS interface to disarm at all: ThrusterController only disarms in destroy_node().
Root cause
Recovery exists as a subtree (SurfaceSafely) but the executor's out-of-band exits (timeout, tick exception, MAX_TICKS) bypass it.
Failure scenario
Run window expires with the sub at 2 m: it sits there, armed, holding depth, until someone toggles the kill switch — burning the surfacing points and looking like a dead vehicle to the judges.
Recommended fix
On any terminal path: command a closed-loop surface (emerge until depth < 0.2 or 20 s), then publish a dedicated disarm command — add one to MovementCommand (or a std_srvs Trigger on the thruster node) so the mission layer can actually release the vehicle.

F12 · Bottom camera: BGRA swap unfixed, default model missing, and a conflicting depth publisher

HighVisionsrc/vision/vision/bottom_camera_node.py:36,144,55Confidence: High
Description
(a) Line 144 still uses COLOR_RGBA2RGB on BGRA frames — the fix in 3ac71152 touched only detector.py, so the bottom camera stays channel-swapped and un-letterboxed even after the pull (its own _inference_thread duplicates the old code). (b) The default model dfc_rs_26.onnx does not exist anywhere in the repo → model load fails → exit_signal=True → the node dies at startup. (c) It publishes depth/sub_depth, the same topic as the front detector, with a different tracking origin — if both cameras run, subscribers see two interleaved depth references.

Failure scenario

Recommended fix
Refactor to share detector.py's fixed inference thread (one implementation), deploy/rename the bottom model, and namespace the topics (depth/sub_depth_front vs …_bottom) or drop the bottom depth publisher entirely — Bar02 is the depth authority.

F13 · Depth-safety response latches full ascent, then stops supervising

HighSafety / Controlautonomous_controller.py:460–465Confidence: High
Description
When depth exceeds 4.5 m the controller sends one emerge 0.6 (duration 0 = "until next command"), sets mode to idle, and returns. Idle mode sends nothing, so the thruster holds 0.6 up-thrust until its own 30 s watchdog fires — an unsupervised full-speed ascent, likely a surface breach at speed.
Root cause
Safety action written as a one-shot instead of a supervised state.
Failure scenario
ZED depth glitches past 4.5 m for one tick → vehicle rockets to the surface for 30 s regardless of the actual depth, ruining the run (and DQ-risk near other vehicles/divers).
Recommended fix
Make an explicit surfacing mode: closed-loop moderate ascent, exit when depth < 0.3 m, then stop. Trip it only after N consecutive over-depth ticks to reject sensor glitches.

F14 · No integrated bring-up: the BT launch starts only itself, and the tree's stubs default to success

HighArchitecture / Launchsrc/robosub2026/launch/shrub.launch.py · safety_nodes.cpp · manipulation_nodes.cppConfidence: High
Description
shrub.launch.py launches bt_executor alone; thruster_node, detector, safety_monitor, depth_node, localization and autonomous_controller (which the BT's nav() calls depend on) must be started by hand in the right order. Scaffold-first development; integration deferred.
Failure scenario

Recommended fix
One mission.launch.py that brings up thruster (simulate:=false explicitly), safety_monitor (simulate:=false), detector, depth_node, autonomous_controller, then the executor. F15 · Blocking MAVLink reads inside timer callbacks stall the 10 Hz control loop

MediumConcurrency / Timingthruster_node.py:201–227, 243–271, 144–199Confidence: High
Description
_arm_vehicle() (called from the armed-check timer) blocks up to 3 s on recv_match; _reconnect_mavlink() (called from the control loop's error path) blocks ~5 s per candidate port on wait_heartbeat. Node callbacks share one mutually-exclusive callback group, so manual_control streaming and the GCS heartbeat stop for the duration.
Failure scenario
During a mid-run re-arm or reconnect the vehicle receives no axis frames for seconds: ALT_HOLD coasts (benign) or, if pilot-input failsafe is enabled, ArduSub reacts to the stream gap. Turns/depth corrections in flight simply freeze.
Recommended fix
Do arming/reconnect on a worker thread (or a Reentrant group + dedicated thread), keep the control-loop callback non-blocking, and feed neutral frames while reconnecting.

F16 · COMMAND_ACK matched without checking which command it acknowledges

MediumPixhawk / MAVLinkthruster_node.py:176,211 · depth_hold_bar02_test.py:454,477Confidence: High
Description
Arm/mode ACKs are read as "next COMMAND_ACK within timeout" — an ack for an unrelated command (stream request, param op, a queued earlier command) satisfies the check. motor_test.py does it correctly (ack.command == MAV_CMD_DO_MOTOR_TEST); the others don't.
Failure scenario
Arm reported "accepted" from a stale ack while the actual arm was rejected by a prearm check → mission proceeds against a disarmed vehicle until the armed-status timer notices (up to 5 s later, then fights it, see F5).
Recommended fix
Filter on ack.command everywhere; treat mismatched acks as noise and keep waiting.

F17 · Dropper's blocking reads will race the Bar02 streamer when integrated into a mission

MediumConcurrency / MAVLinkdropper.py:49–99Confidence: Medium
Description
Dropper._recv/_servo_raw/_get_param call recv_match(blocking=True) on the shared master. In the DepthKeeper missions, Bar02DepthSource claims the serial recv path (master._external_recv_reader) and reads from a background thread — a second blocking reader reproduces the documented two-reader pyserial race ("readiness to read but returned no data") and steals the streamer's pressure/attitude messages.
Failure scenario
Bin task: mission calls dropper.prepare()/drop_right() mid-run → depth/attitude stream hiccups, keeper logs "no Bar02 depth", possibly a serial exception storm during the drop — at the exact moment the vehicle must hover steady.
Recommended fix
Give Dropper a passive mode when master._external_recv_reader is set: read PARAM_VALUE / SERVO_OUTPUT_RAW / COMMAND_ACK from master.messages (timestamp-gated) instead of recv_match, mirroring the thruster node's _passive_heartbeat().

F18 · Fused localization is frame-inconsistent: Z-up yaw math on Y-up data, two VIO sources, spliced depth axis

MediumCoordinate framessrc/localization/localization/localization_node.py:30–43, 80–88, 174–181Confidence: High
Description
(a) _quat_to_yaw uses the Z-up yaw formula, but both odometry sources are ZED RIGHT_HANDED_Y_UP (vertical = Y; field_common correctly computes heading about Y). The fused yaw — consumed by autonomous_controller and MissionIO with the same Z-up assumption — is wrong whenever the camera pitches/rolls. (b) odom/bottom and vslam/odometry feed one callback: two live sources interleave and the pose jumps between two origins. (c) Depth is written into pose.z while x/y remain camera-world axes — a hybrid frame no consumer documents.
Failure scenario
waypoint/station_keep (already broken by F9) get a heading that swings with pitch; with both cameras running the anchor point teleports; drift corrections computed in the hybrid frame push the fused pose further off.
Recommended fix
Pick one world convention (suggest: convert ZED Y-up → ROS ENU/Z-up at each publisher), one VIO source per run (parameter), and document localization/pose's frame in the message header. Given VSLAM is deliberately disabled for competition, consider deleting this path until it's needed — it currently only misleads.

F19 · Detection ranges attach to the wrong objects when the ZED tracker reorders

MediumVision / ZEDdetector.py:601–607 (enrich_depths) · bottom_camera_node.py:319–325Confidence: Medium
Description
enrich_depths zips objects.object_list[i] onto detection_infos[i] by index. The ZED object tracker does not preserve ingest order — it drops unconfirmed boxes and appends persistent tracks — so ranges can land on the wrong labels whenever more than one object is in view.
Failure scenario
Gate + marker both visible: the gate detection gets the marker's 1.2 m range → AlignGateCenter's in_range fires early → transit starts from the wrong standoff.
Recommended fix
Match by geometry: for each tracked object, take its 2D bbox (bounding_box_2d) and pair with the detection of maximum IoU; leave range −1 when no match clears a threshold.

F20 · Battery monitor: unknown voltage reads as 100%, and simulate defaults on

MediumSafety / Sensorssafety_monitor_node.py:53,126–133Confidence: High
Description
(a) When battery_remaining = −1, the fallback divides voltage_battery by 1000; MAVLink uses UINT16_MAX (65535) for "unknown", which becomes 65.5 V → curve clamps to 100%. (b) simulate defaults true, publishing a flat 100% — if the comp launch forgets simulate:=false + an endpoint, the BT's battery failsafe is permanently blind. (c) Leak detection is a stub returning false (no hardware yet — flagged so it isn't mistaken for coverage).
Failure scenario
Pack sags to 13 V mid-run with BATT_CAPACITY unset and voltage reporting unknown → dashboard shows 100% → no critical_failure → brownout instead of a controlled surface.
Recommended fix
Treat voltage_battery == 65535 (and 0) as unknown → NaN; default simulate:=false and fail loudly when no endpoint is configured.

F21 · Control threads have no exception guard — one bad publish silently kills motion control

MediumConcurrency / Safetyfield_common.py:193–208 (_stream_loop) · origin DepthKeeper._loopConfidence: High
Description
RampedDriver's streamer thread and DepthKeeper's control thread run bare loops: any exception (publish after context teardown, transient rclpy error, the F2 AttributeError) terminates the daemon thread with no log and no fallback.
Failure scenario
Keeper dies mid-leg → thruster holds the last surge/heave for up to 30 s (watchdog), driving blind into the gate structure, then holds depth armed. Nothing on screen explains why the mission froze.
Recommended fix
Wrap loop bodies in try/except: log loudly, publish a stop, retry a few times, and set a dead flag the mission can poll (session teardown should also check it).

F22 · Thruster watchdog is 30 s and stops without disarming

MediumSafety / Thrustersthruster_node.py:55, 492–501Confidence: High

F24 · BT motion primitives are uncalibrated open-loop constants; YawSweep doesn't do what its name says

4.2 Safety-critical findings

F4, F5, F6, F11, F13, F20, F22 — plus the structural fact that no software path can disarm the vehicle on request; disarm happens only on process teardown. The kill switch is the real safety system today; the software should stop assuming it will always be reachable.

4.3 Pixhawk / MAVLink

Connection handling (F6, F15, F16), re-arm policy (F5), ack discipline (F16), the two-reader serial rule (F17 — correctly handled between ThrusterController and Bar02DepthSource on origin via _external_recv_reader + passive master.messages reads; Dropper is the remaining violator). The pymavlink 2.4.49 add_message crash is monkeypatched in depth_hold_bar02_test and (via import) everything that uses it; standalone scripts (motor_test, dropper) carry their own TypeError guards — keep that pattern for any new script.

4.4 Thruster control

Axis clamping and NaN guarding in ThrusterController are genuinely good (finite checks, type checks, clamped ints, neutralize-on-garbage). Remaining issues: deadzone mapping (F10), stale-axis verbs (F9), watchdog policy (F22), and the fact that stop() uses z=500 which in MANUAL means "no vertical thrust" — correct, but callers should know depth is then buoyancy-determined.

4.5 ROS communication & topic graph

Graph as-built: detector → vision/detections, depth/sub_depth, vslam/odometry; bottom camera → odom/bottom, vision/path_markers, depth/sub_depth (collision, F12); depth_node → depth/info; localization → localization/pose; controller/BT/scripts → movement_command, navigation_command. Multiple-writer exposure on movement_command and depth/sub_depth is the main graph defect. Orphans: vslam/path, localization/correction (publisher exists, but drift correction depends on the dead bottom camera), BehaviorStatus.msg (unused). QoS is uniformly default depth-10 reliable — fine at these rates.

4.6 TF & frames

Only vslam_node broadcasts TF (odom → zed_left_camera_frame); nothing consumes TF. All frame handling is ad-hoc per node — the Y-up/Z-up split (F18) is the systemic issue. No IMU→camera→base_link extrinsics anywhere; acceptable while control is image-space reactive, blocking for any world-frame ambition.

4.7 ZED SDK

Open/retry/reopen loops are well built (candidate ladders, close+gc on failure, SIGHUP handlers to release the camera — clearly hard-won). Gaps: tracking-state ignored (F8), one-ZED-per-process rule enforced only by docstring, service-thread SDK calls unsynchronized (F26), and grab→retrieve→get_data buffer reuse is lock-protected correctly in both camera nodes.

4.8 IMU / barometer / odometry / fusion

Bar02: detection, 30BA-misdetect sanity gate, and surface latching are excellent; staleness handling exists in DepthKeeper but not the standalone test (F4). ATTITUDE-based gyro fallback for turns is smart; its |rate| integration is not (F7). ZED odometry: state-gating (F8) and the "front camera as odometry source" shortcut are the weak points. Fusion (localization_node) is an alpha-blend offset scheme with frame bugs (F18) — currently unused by the proven mission path, which is the right call.

4.9 Computer vision

The native TensorRT path (cuda-python, pre-allocated buffers, single stream, thread-safe wrapper, engine cached on disk) is solid engineering for the Orin. Issues: preprocessing bugs (F3/F12), index-based depth enrichment (F19), _parse_yolo's shape heuristic is fine for current exports but will misparse an end2end export with >300 anchors of 6 channels (document the contract), and NMS is O(n²) capped at 500 — fine. Engine files exist for ffc_rs_26; the bottom model doesn't (F12).

4.10 State machines / BT

The Python BT engine (bt.py) is clean. SHRUB (C++) structure is sound (tick loop, timeout, exception guard, blackboard seeding) but the leaves are scaffolds (F14, F24); the mission timeout epilogue is unsafe (F11). The prequalification FSM is the most battle-tested mission logic in the repo — mine it for timeout/fallback patterns when filling in SHRUB.

4.11 Timing, concurrency, memory, performance

Blocking-in-callback (F15), unguarded threads (F21), busy-ish loops all sleep properly. Memory: CUDA buffers freed in __del__ best-effort; sl.Mat reuse is right; Path capped at 500 poses; log tee fsyncs every chunk (deliberate, power-cut-proof — costs some IO). CPU: detector at 416² TRT ~37 fps is comfortable; the --view path correctly off by default. No leaks of consequence found in project code.

4.12 Configuration & parameters

PID live-tuning via parameters is well done. Gaps: safety_monitor's simulate-by-default (F20), BT magic numbers (F24), THR_DZ hardcoded (F10), TURN_90_SECONDS/ROTATE_90_SECONDS calibration constants shared by several scripts but owned by none, and the known param-drift problem (MOT_x_DIRECTION incident) — keep diffing live params against pixhawk_params_4.5.7_backup_2026-07-08.param before every water day; consider automating that diff in boot_check.py.

4.13 Testing gaps

Only colcon lint templates exist. Highest-value additions, in order: (1) a message-compat smoke test (catches F2 class instantly); (2) unit tests for centering.py and _parse_yolo/letterbox round-trip (pure functions, easy); (3) a signed-rotation test for the turn integrators once F7 is fixed; (4) a bench "arm-check" script asserting mode/arm/ack discipline against the real Pixhawk; (5) SITL (ArduSub SIM) run of gate_task before each competition day.

4.14 Security

Threat surface is physical/serial, not network: no exposed services, no shell injection paths found; deploy_model.sh/setup scripts are local-only. MAVLink is unsigned but that's standard for a tethered/local link. No action needed beyond not bridging the ROS domain to hotel Wi-Fi.

4.15 Code quality & refactoring

Working-tree Python is consistently documented (docstrings encode field lessons — genuinely valuable). Priorities: unify the two inference threads (F12), extract one shared MAVLink helper (connect/ack/param-set logic exists in five copies with different bug levels), retire legacy runners (F25), and converge on one depth authority and one heading convention. The C++ BT code is tidy; its problem is emptiness, not quality.
